================================================================================
  DATA FLOW DIAGRAMS
================================================================================

1. INTAKE SESSION FLOW
   ─────────────────────

   Client Opens /intake
        │
        ▼
   Frontend fetches intake flow (GET /api/intake/flow)
        │
        ▼
   Display questions step-by-step
        │
        ▼
   Client answers question
        │
        ▼
   Frontend submits (POST /api/intake/step)
        │
        ▼
   Backend validates & stores in DB
        │
        ▼
   Frontend renders next question
        │
        ▼
   Repeat until complete
        │
        ▼
   Client completes intake (POST /api/intake/complete)
        │
        ▼
   Backend marks session as complete
        │
        ▼
   Lawyer can view in dashboard


2. AI SUMMARY GENERATION
   ──────────────────────

   Lawyer clicks "Generate Summary"
        │
        ▼
   Frontend: POST /api/summary/generate {session_id}
        │
        ▼
   Backend SummaryService:
        │
        ├─ Fetch all messages from session
        │
        ├─ Extract text from uploaded PDFs (PDFService)
        │
        ├─ Build structured prompt
        │
        ├─ Send to Ollama HTTP API
        │
        ├─ Parse JSON response
        │
        └─ Store in DB (ai_summary JSONB field)
        │
        ▼
   Frontend displays summary with:
        ├─ Case summary
        ├─ Legal category
        ├─ Urgency level
        ├─ Key facts
        ├─ Missing information
        └─ Recommended next questions


3. DOCUMENT UPLOAD FLOW
   ────────────────────

   Client uploads PDF during intake
        │
        ▼
   Frontend: POST /api/files/upload (multipart/form-data)
        │
        ▼
   Backend validates file:
        ├─ Check MIME type
        ├─ Check file size (<50MB)
        └─ Scan for malware (optional)
        │
        ▼
   Backend uploads to Supabase Storage
        │
        ▼
   Backend creates file record in DB
        │
        ▼
   Backend extracts text (PyMuPDF)
        │
        ▼
   Text stored in messages table for AI context
        │
        ▼
   Frontend displays file in session

================================================================================
  SCALING ROADMAP
================================================================================

MVP (Current):
  • Single backend instance
  • Local Ollama
  • Managed Supabase
  • Suitable for 100-1000 users

Phase 2 (1000-10000 users):
  • Load balancer
  • Multiple backend instances
  • Database read replicas
  • Redis caching layer

Phase 3 (10000+ users):
  • Kubernetes orchestration
  • Ollama API service
  • CDN for static assets
  • Message queue (Celery)
  • Microservices architecture
